# EDA 구역 제약 조건 (Exploratory Data Analysis Constraints)

## 1. 대용량 데이터 메모리 제약 (Memory Management)
* 영수증(Transaction) 데이터 통제: 수백만 건 이상의 결제 데이터를 pandas로 한 번에 로드하지 말고, 다운캐스팅, Chunk 읽기 또는 Polars를 활용해 OOM 에러를 방지하십시오.
* 중간 산출물은 가급적 `Parquet` 포맷을 사용하십시오.

## 2. 원본 데이터 절대 보존 (Strict Read-Only)
* `data/raw/` 및 `data/processed/` 디렉토리는 절대 수정하지 마십시오.
* 전처리가 완료된 데이터는 `data/processed/` 디렉토리에 분리하여 저장하십시오.

## 3. 그래프 구조 친화적 분석 (Graph-Oriented EDA)
* 노드 희소성(Sparsity): 출현 빈도가 극히 낮은 속성/트렌드 노드의 분포를 파악하고 시각화하여 가지치기(Pruning) 기준을 세우십시오.
* 연결 밀도: 영수증 당 평균 상품 수(Degree distribution) 등 이분 그래프(Bipartite) 특성을 분석하십시오.

## 4. 시각화 강제 규칙
* matplotlib, seaborn, plotly 사용 시 반드시 차트 제목, X/Y 축 라벨, 범례를 명시하여 가독성을 확보하십시오.

## 5. 분석 결과물 관리 규칙
* 모든 EDA 분석 결과물은 `eda/ipynb/` 폴더 내에 `.ipynb` (Jupyter Notebook) 형식으로 작성하고 관리하십시오.
* 개별 파이썬 스크립트(`.py`)보다는 실험 과정과 시각화가 포함된 노트북 형식을 우선합니다.

## 6. 코드 전달 및 문서화 방식 (Code Delivery & Documentation)
* 모든 분석 로직은 `eda/` 폴더 내에 새로운 `.py` 파일로 생성하여 제공한다.
* 분석 가설, 상세 설명, 통계 방법론 등은 해당 `.py` 파일 내에 상세한 주석으로 기술한다.
* 채팅에서는 분석의 핵심 전략과 비즈니스 인사이트 요약만을 전달하여 문맥 효율성을 극대화한다.
* 사용자는 생성된 `.py` 파일의 내용을 복사하여 Jupyter Notebook 셀에 붙여넣어 실행한다.

## 7. 설명 후 구현 원칙 (Explain Before Implementation)
* 모든 코드 작성 전, 해당 로직의 수학적/통계적 근거와 비즈니스 논리를 텍스트로 먼저 상세히 설명한다.
* 사용자가 설명에 동의하고 "코드를 작성해 달라"고 명시적으로 요청한 경우에만 `.py` 파일 생성 및 코드를 제공한다.
