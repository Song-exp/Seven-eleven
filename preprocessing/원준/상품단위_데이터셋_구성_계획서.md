# 상품 단위 데이터셋 구성 계획서

## 목적

**원본 데이터셋**: `pos_data_food_final_상품단위변환전.parquet`  
**대상 데이터셋**: `pos_data_food_final_상품단위.csv`  
**목표**: 거래 단위(행 = 1건의 판매) → **상품 단위(행 = 1개 상품)** 로 집계하여, 상품 성공 여부 분류 모델의 학습 데이터 구축

---

## ⚠️ 핵심 설계 원칙: 메모리 효율화

원본 데이터셋은 약 **7천만 행** 규모이므로 메모리 에러를 방지하는 것이 최우선 과제입니다.

### 메모리 절약 전략

| 전략 | 적용 방법 |
|---|---|
| **필요 컬럼만 로드** | `pd.read_parquet(..., columns=[...])` 로 사용할 컬럼만 선택적 로드 |
| **데이터타입 다운캐스팅** | `매출금액`, `매출수량` → `float32` / 날짜 관련 → `category` or `int16` |
| **중간 결과 즉시 삭제** | 조기 매출 필터링용 서브셋 사용 후 `del` + `gc.collect()` |
| **조기 매출 피처 분리 계산** | 기본 집계(groupby)와 조기 매출 계산을 따로 분리하여 메모리 피크 분산 |
| **카테고리 컬럼 최적화** | 중/소분류명 등 반복값 많은 컬럼 → `.astype('category')` |

## 상품 단위 데이터셋 컬럼 설계

### 1. 식별자 / 카테고리 컬럼 (기본 정보)

| 컬럼명 | 도출 방법 | 설명 |
|---|---|---|
| `상품코드` | groupby 기준 | 상품 고유 식별자 |
| `상품명` | first() | 상품 이름 (동일 코드 → 대표명) |
| `상품대분류명` | `.str.strip()` 후 first() | 대분류 |
| `상품중분류명` | `.str.strip()` 후 first() | 중분류 |
| `상품소분류명` | `.str.strip()` 후 first() | 소분류 |

> **strip() 필수 적용**: 대/중/소분류명 컬럼은 groupby 또는 집계 전에 반드시 `.str.strip()`을 적용하여 불필요한 공백으로 인한 중복 카테고리 생성을 방지

---

### 2. 기간 기준 컬럼

| 컬럼명 | 도출 방법 | 설명 |
|---|---|---|
| `첫_판매일` | `판매시간_dt.min()` | 해당 상품이 처음 판매된 날짜 |
| `마지막_판매일` | `판매시간_dt.max()` | 해당 상품이 마지막으로 판매된 날짜 |
| `판매_지속기간(일)` | `max - min` | 첫 판매일부터 마지막 판매일까지의 일수 |
| `판매_활성_월수` | 판매된 고유 월 수 | 몇 개월 동안 팔렸는지 |

> **설계 근거**: 계절상품, 단종상품, 지속판매 상품을 구분하는 데 핵심. 성공 상품은 긴 판매 지속기간과 많은 활성 월수를 가지는 경향이 있음.

### 3. 거래 규모 컬럼 (집계 피처)

| 컬럼명 | 도출 방법 | 설명 |
|---|---|---|
| `총_매출수량` | `매출수량.sum()` | 전체 기간 판매 수량 합계 |
| `총_매출금액` | `매출금액.sum()` | 전체 기간 판매 금액 합계 |
| `객단가` | `총_매출금액 / 총_매출수량` | 상품 평균 단가 |
| `총_거래건수` | `count()` | 해당 상품이 포함된 거래 건수 |
| `취급_점포수` | `점포코드.nunique()` | 몇 개 점포에서 판매됐는지 |

> **설계 근거**: 성공 상품의 핵심 지표. 소수 점포에서만 팔린 상품 vs 전국 모든 점포에서 팔린 상품의 성공 가능성은 크게 다름.

### 4. 조기 매출 트렌드 컬럼 (출시 초기 성과)

> **핵심 피처**: 상품 성공 여부는 출시 초기 반응으로 예측 가능. 조기 매출 데이터가 모델의 핵심 독립변수 역할을 함.

| 컬럼명 | 도출 방법 | 설명 |
|---|---|---|
| `출시후_3일_매출액_합계` | `첫_판매일 + 3일 이내` 거래의 `매출금액.sum()` | 출시 직후 3일간 총 매출 |
| `출시후_7일_매출액_합계` | `첫_판매일 + 7일 이내` | 출시 후 1주일간 총 매출 |
| `출시후_30일_매출액_합계` | `첫_판매일 + 30일 이내` | 출시 후 1개월간 총 매출 |
| `출시후_3일_매출액_평균` | 위 합계 / 3 | 출시 후 3일 일평균 매출 |
| `출시후_7일_매출액_평균` | 위 합계 / 7 | 출시 후 7일 일평균 매출 |
| `출시후_30일_매출액_평균` | 위 합계 / 30 | 출시 후 30일 일평균 매출 |
| `출시후_3일_매출수량_합계` | `첫_판매일 + 3일 이내` 거래의 `매출수량.sum()` | 출시 직후 3일간 총 판매 수량 |
| `출시후_7일_매출수량_합계` | `첫_판매일 + 7일 이내` | 출시 후 1주일간 총 판매 수량 |
| `출시후_30일_매출수량_합계` | `첫_판매일 + 30일 이내` | 출시 후 1개월간 총 판매 수량 |

#### 관측 기간 경계 처리 규칙 (12월 출시 상품)

관측 기간이 **2025년 1년**으로 한정되므로, 출시 시점에 따라 다음 규칙을 적용합니다.

| 조기 매출 피처 | 관측 불가 조건 | 처리 방법 |
|---|---|---|
| `출시후_3일_*` | `첫_판매일 > 2025-12-29` | 관측 가능 일수만큼 채우되, 완전 관측(3일) 불가 시 → `NaN` |
| `출시후_7일_*` | `첫_판매일 > 2025-12-25` | 관측 가능 일수만큼 채우되, 완전 관측(7일) 불가 시 → `NaN` |
| `출시후_30일_*` | `첫_판매일 > 2025-12-02` | 관측 가능 일수만큼 채우되, 완전 관측(30일) 불가 시 → `NaN` |

> **예시**: 12월 27일 출시 상품의 경우
> - `출시후_3일_*` → 12/27, 12/28, 12/29: **3일 채울 수 없어 NaN**
> - `출시후_7일_*` → 관측 불가 → `NaN`
> - `출시후_30일_*` → 관측 불가 → `NaN`
>
> **예시**: 12월 1일 출시 상품의 경우
> - `출시후_3일_*` → 12/1 ~ 12/3: **완전 관측 가능**, 값 채움
> - `출시후_7일_*` → 12/1 ~ 12/7: **완전 관측 가능**, 값 채움
> - `출시후_30일_*` → 12/1 ~ 12/30: **완전 관측 가능**, 값 채움

#### 메모리 효율적 계산 방법

조기 매출 피처를 계산할 때 전체 7천만 행을 한번에 처리하면 메모리 에러 발생 가능. 아래 방식으로 분리 계산:

```python
# 전체 df에서 필요한 컬럼만 로드 (판매시간_dt, 상품코드, 매출금액, 매출수량)
# 상품별 첫 판매일 사전(dict) 생성 → join 없이 직접 참조
# 조기 매출 필터링 → groupby 집계 → 기본 집계 테이블에 merge
# 필터링 서브셋은 merge 후 즉시 del + gc.collect()
```

## 구현 순서 (다음 노트북 작업 계획)

1. **로드**: `pos_data_food_final_상품단위변환전.parquet`에서 **필요한 컬럼만** 선택적 로드
2. **전처리**: 대/중/소분류명 `.str.strip()` 적용 → 불필요한 공백 제거
3. **기본 집계**: `상품코드` 기준 groupby → 기간/규모/패턴 피처 일괄 집계
4. **조기 매출 피처**: 상품별 첫 판매일 기준으로 3/7/30일 필터링 → 별도 집계 후 merge
   - 완전 관측 불가 상품(해당 기간이 연도 경계를 벗어남) → `NaN`
   - merge 후 임시 서브셋 `del` + `gc.collect()`
5. **저장**: `pos_data_food_final_상품단위.csv` (encoding: utf-8-sig)
